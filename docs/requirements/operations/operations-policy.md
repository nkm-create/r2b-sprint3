# 運用方針定義

## 1. 概要

本ドキュメントは、学習塾時間割最適化システムの開発・運用体制を定義する。

---

## 2. チーム構成

### 2.1 開発チーム

| 役割 | 人数 | 担当範囲 |
|------|------|----------|
| プロダクトオーナー | 1名 | 要件定義、優先度決定、受け入れ判断 |
| テックリード | 1名 | アーキテクチャ設計、技術選定、コードレビュー |
| フロントエンドエンジニア | 2名 | UI/UX実装、React開発 |
| バックエンドエンジニア | 2名 | API実装、DB設計、最適化エンジン |
| QAエンジニア | 1名 | テスト計画、E2Eテスト、品質管理 |

### 2.2 運用チーム

| 役割 | 人数 | 担当範囲 |
|------|------|----------|
| インフラエンジニア | 1名 | クラウド運用、監視、障害対応 |
| サポート担当 | 1名 | ユーザー問い合わせ対応、マニュアル整備 |

### 2.3 ステークホルダー

| 役割 | 所属 | 関与 |
|------|------|------|
| システム管理者 | 本部 情報システム部門 | ユーザー管理、教室マスタ管理 |
| エリアマネジャー | 各エリア | 複数教室の充足率モニタリング |
| 教室長 | 各教室 | 時間割作成・管理の実務 |

## 3. ブランチ戦略

### 3.1 ブランチモデル

**GitHub Flow** を採用する。

```
main ─────────────────────────────────────────────────→
      ↑          ↑          ↑          ↑
      │          │          │          │
   feature/   feature/   feature/   hotfix/
   ISSUE-001  ISSUE-002  ISSUE-003  ISSUE-004
```

### 3.2 ブランチ命名規則

| ブランチ種別 | 命名パターン | 用途 |
|-------------|-------------|------|
| main | `main` | 本番環境にデプロイされるコード |
| feature | `feature/ISSUE-XXX-description` | 新機能開発 |
| bugfix | `bugfix/ISSUE-XXX-description` | バグ修正 |
| hotfix | `hotfix/ISSUE-XXX-description` | 本番緊急修正 |
| release | `release/vX.Y.Z` | リリース準備（必要に応じて） |

### 3.3 ブランチ運用ルール

1. **main ブランチへの直接コミット禁止**
   - 全ての変更はプルリクエスト経由
   - CI/CDパイプラインの成功が必須

2. **プルリクエストルール**
   - 最低1名のコードレビュー承認が必要
   - テストカバレッジ80%以上を維持
   - Linter/Formatter チェック通過必須

3. **コミットメッセージ規約**
   - Conventional Commits 形式を採用
   ```
   <type>(<scope>): <subject>

   例:
   feat(schedule): add optimization algorithm
   fix(teacher): correct validation logic
   docs(api): update endpoint documentation
   ```

### 3.4 コミットタイプ

| タイプ | 説明 |
|--------|------|
| feat | 新機能 |
| fix | バグ修正 |
| docs | ドキュメント変更 |
| style | コードフォーマット変更 |
| refactor | リファクタリング |
| test | テスト追加・修正 |
| chore | ビルド・補助ツールの変更 |

## 4. クラウド選定

### 4.1 クラウドプロバイダー

**Amazon Web Services (AWS)** を採用する。

### 4.2 選定理由

| 観点 | AWS選定理由 |
|------|-------------|
| 実績 | 国内学習塾・教育系システムでの採用実績が豊富 |
| サービス充実度 | 必要なマネージドサービスが揃っている |
| コスト | 従量課金により初期費用を抑制可能 |
| 国内リージョン | 東京リージョンでの低レイテンシ |
| サポート | 日本語サポートが充実 |

### 4.3 利用サービス構成

| カテゴリ | サービス | 用途 |
|----------|----------|------|
| コンピュート | ECS Fargate | アプリケーション実行基盤 |
| コンテナレジストリ | ECR | Dockerイメージ管理 |
| データベース | RDS (PostgreSQL) | リレーショナルデータベース |
| キャッシュ | ElastiCache (Redis) | セッション管理、キャッシュ |
| ストレージ | S3 | 静的ファイル、バックアップ |
| CDN | CloudFront | 静的アセット配信 |
| ロードバランサー | ALB | トラフィック分散 |
| DNS | Route 53 | ドメイン管理 |
| シークレット管理 | Secrets Manager | 認証情報・API鍵管理 |
| ログ管理 | CloudWatch Logs | ログ集約・検索 |
| モニタリング | CloudWatch | メトリクス監視・アラート |
| 非同期処理 | SQS | メッセージキュー |
| 証明書 | ACM | SSL/TLS証明書管理 |

## 5. リリースフロー

### 5.1 デプロイ戦略

**Blue-Green デプロイ** を採用する。

```
┌─────────────────────────────────────────────┐
│                    ALB                       │
│              (トラフィック振り分け)            │
└─────────────────────────────────────────────┘
         │                      │
         ▼                      ▼
┌──────────────────┐    ┌──────────────────┐
│   Blue 環境       │    │   Green 環境      │
│   (現行バージョン)  │    │   (新バージョン)   │
│   100% ──────────│────│→ 0%             │
│   トラフィック      │    │                  │
└──────────────────┘    └──────────────────┘
         │                      │
    デプロイ後              段階的に
                         トラフィック移行
```

### 5.2 リリースプロセス

```
┌────────────────────────────────────────────────────────────────┐
│  1. 開発完了                                                    │
│     - feature ブランチでの実装完了                               │
│     - ローカルテスト通過                                         │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  2. プルリクエスト                                               │
│     - コードレビュー依頼                                         │
│     - CI パイプライン実行（Lint、テスト、ビルド）                  │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  3. コードレビュー & 承認                                        │
│     - 最低1名のレビュアー承認                                    │
│     - 修正依頼への対応                                           │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  4. main マージ                                                 │
│     - Squash merge で履歴を整理                                  │
│     - マージ後、自動でステージング環境にデプロイ                   │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  5. ステージング検証                                             │
│     - QAエンジニアによる動作確認                                  │
│     - E2Eテスト実行                                             │
│     - 性能テスト（必要に応じて）                                  │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  6. 本番リリース承認                                             │
│     - プロダクトオーナーによる承認                                │
│     - リリースノート作成                                         │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  7. 本番デプロイ                                                 │
│     - Blue-Green デプロイ実行                                    │
│     - ヘルスチェック確認                                         │
│     - 段階的トラフィック切り替え（10% → 50% → 100%）              │
└────────────────────────────────────────────────────────────────┘
                              ▼
┌────────────────────────────────────────────────────────────────┐
│  8. リリース完了                                                 │
│     - 監視ダッシュボードでの確認                                  │
│     - 旧環境の保持（ロールバック用）                              │
│     - リリースタグ作成                                           │
└────────────────────────────────────────────────────────────────┘
```

### 5.3 CI/CD パイプライン

**GitHub Actions** を使用する。

#### パイプライン構成

```yaml
# .github/workflows/ci.yml
name: CI Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    # コードスタイルチェック

  test:
    # ユニットテスト・結合テスト

  build:
    # Dockerイメージビルド

  deploy-staging:
    # ステージング環境デプロイ（main マージ時のみ）

  deploy-production:
    # 本番環境デプロイ（手動トリガー）
```

#### ステージ別実行内容

| ステージ | 実行内容 | 実行条件 |
|----------|----------|----------|
| lint | ESLint, Prettier, TypeScript型チェック | 全PR |
| test | Jest単体テスト、結合テスト | 全PR |
| build | Dockerイメージビルド、ECRプッシュ | mainマージ後 |
| deploy-staging | ステージング環境デプロイ | mainマージ後（自動） |
| deploy-production | 本番環境デプロイ | 手動承認後 |

### 5.4 ロールバック手順

1. **即座のロールバック**（5分以内）
   - ALBのターゲットグループを旧環境に切り替え
   - 新環境を停止

2. **データベース変更を伴う場合**
   - マイグレーションのロールバックスクリプト実行
   - データ整合性の確認

3. **ロールバック判断基準**
   - エラー率が5%を超えた場合
   - レスポンスタイムが3秒を超えた場合
   - 重大なセキュリティ問題が発見された場合

## 6. リリーススケジュール

### 6.1 定期リリース

| 項目 | 内容 |
|------|------|
| リリース頻度 | 2週間に1回（スプリント単位） |
| リリース曜日 | 火曜日 |
| リリース時間帯 | 10:00〜12:00 |
| 禁止期間 | 金曜日、祝前日、ターム切替期間（3月・9月） |

### 6.2 緊急リリース

| 項目 | 内容 |
|------|------|
| 対象 | セキュリティ脆弱性、重大なバグ |
| 承認者 | テックリード + プロダクトオーナー |
| 手順 | hotfixブランチから直接本番デプロイ |

## 7. 監視・アラート運用

### 7.1 監視対象

| カテゴリ | メトリクス | 閾値 |
|----------|-----------|------|
| インフラ | CPU使用率 | 80%以上で警告、90%以上で重大 |
| インフラ | メモリ使用率 | 80%以上で警告、90%以上で重大 |
| インフラ | ディスク使用率 | 70%以上で警告、85%以上で重大 |
| アプリ | エラー率 | 1%以上で警告、5%以上で重大 |
| アプリ | レスポンスタイム（P95） | 2秒以上で警告、3秒以上で重大 |
| DB | コネクション数 | 80%以上で警告 |
| DB | レプリケーション遅延 | 10秒以上で警告 |

### 7.2 アラート通知

| 重要度 | 通知先 | 対応時間 |
|--------|--------|----------|
| 重大 | Slack #alerts + PagerDuty（電話） | 即時対応 |
| 警告 | Slack #alerts | 1時間以内に確認 |
| 情報 | Slack #monitoring | 翌営業日確認 |

### 7.3 オンコール体制

| 項目 | 内容 |
|------|------|
| 体制 | 週替わりの1名体制 |
| 対象時間 | 平日 9:00〜22:00 |
| 深夜・休日 | 重大アラートのみ自動エスカレーション |

## 8. セキュリティ運用

### 8.1 脆弱性管理

| 項目 | 内容 |
|------|------|
| 依存パッケージスキャン | Dependabot による自動検知 |
| コンテナスキャン | ECRによる自動スキャン |
| 定期診断 | 四半期ごとの脆弱性診断 |

### 8.2 インシデント対応

| レベル | 定義 | 対応時間 |
|--------|------|----------|
| 緊急 | データ漏洩、認証バイパス | 即時対応、1時間以内に報告 |
| 高 | 権限昇格、重要機能の不正利用 | 4時間以内に対応 |
| 中 | 軽微な情報漏洩リスク | 24時間以内に対応 |
| 低 | 潜在的リスク | 次回リリースで対応 |

## 9. バックアップ・リカバリ

### 9.1 バックアップ運用

| 種別 | 頻度 | 保持期間 | 保存先 |
|------|------|----------|--------|
| RDS スナップショット | 日次 | 30日 | 同一リージョン |
| S3 バックアップ | 日次 | 30日 | クロスリージョン |
| 設定ファイル | 変更時 | 無期限 | Git |

### 9.2 リカバリ手順

| 障害種別 | RTO | RPO | 手順概要 |
|----------|-----|-----|----------|
| アプリケーション障害 | 15分 | 0 | ECSタスク再起動 |
| DB障害 | 1時間 | 1時間 | RDSスナップショットからリストア |
| リージョン障害 | 4時間 | 1時間 | DR環境への切り替え |

## 10. ドキュメント管理

### 10.1 ドキュメント種別

| ドキュメント | 管理場所 | 更新タイミング |
|-------------|----------|---------------|
| 設計ドキュメント | Git リポジトリ | 設計変更時 |
| API仕様書 | Git リポジトリ（OpenAPI） | API変更時 |
| 運用手順書 | Notion/Confluence | 手順変更時 |
| 障害報告書 | Notion/Confluence | 障害発生時 |
| ユーザーマニュアル | ヘルプサイト | 機能追加時 |

### 10.2 変更管理

| 項目 | ルール |
|------|--------|
| 設計変更 | プルリクエストでレビュー必須 |
| 運用手順変更 | 変更履歴を記録、関係者への周知必須 |
| 緊急対応 | 事後でも必ずドキュメント化 |
