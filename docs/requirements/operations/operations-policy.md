## 3. 技術スタック

### 3.1 フロントエンド

| カテゴリ | 技術 | 選定理由 |
|----------|------|----------|
| フレームワーク | Next.js 14 (App Router) | SSR/SSG対応、React Server Components |
| 言語 | TypeScript | 型安全性、開発効率 |
| スタイリング | Tailwind CSS | ユーティリティファースト、一貫性 |
| UIコンポーネント | shadcn/ui | カスタマイズ性、アクセシビリティ |

| フォーム | React Hook Form + Zod | バリデーション統合 |
| カレンダー/DnD | FullCalendar + dnd-kit | 時間割表示、ドラッグ&ドロップ |
| グラフ | Recharts | React対応、カスタマイズ性 |

### 3.2 バックエンド

| カテゴリ | 技術 | 選定理由 |
|----------|------|----------|
| フレームワーク | FastAPI | 非同期対応、自動APIドキュメント |
| 言語 | Python 3.11+ | AI/ML連携、豊富なライブラリ |
| ORM | SQLAlchemy 2.0 | 非同期対応、型ヒント |
| バリデーション | Pydantic v2 | 高速、FastAPI統合 |
| タスクキュー | Celery + Redis | 非同期処理、スケジューリング |
| 認証 | JWT + OAuth 2.0 | ステートレス、Google連携 |

### 3.3 AI/最適化

| カテゴリ | 技術 | 選定理由 |
|----------|------|----------|
| LLMフレームワーク | LangChain / LangGraph | エージェントオーケストレーション |
| LLMプロバイダ | **Anthropic Claude 3（推奨）** | 説明生成に優れる、高品質な日本語対応 |
| 最適化ソルバー | Google OR-Tools (CP-SAT) | 制約充足問題、オープンソース |
| ベクトルDB | Pinecone / Qdrant | エージェント記憶（将来拡張） |

#### LLM活用方針（v1.1）

| コンポーネント | LLM活用 | 用途 |
|--------------|--------|------|
| Schedule Orchestrator | ❌ 不使用 | 状態マシンによるソルバー監視・戦略切り替え |
| Analyzer | ❌ 不使用 | ルールベースの問題分析・戦略提案 |
| Solver | ❌ 不使用 | OR-Tools CP-SATによる一発求解 |
| **Explainer** | ✅ 使用 | **結果説明・緩和提案・What-if分析** |

#### LLM非機能要件

| 項目 | 目標値 | 備考 |
|------|--------|------|
| レスポンス時間 | 500ms〜1.5秒 | 結果説明・What-if分析で使用 |
| 可用性 | 99.5%以上 | フォールバック込み |
| 月額コスト上限 | $500以下 | 1000教室想定 |
| タイムアウト | 3秒 | 超過時はフォールバック |

#### フォールバック機構

| 状況 | 対応 |
|------|------|
| LLM呼び出し失敗 | テンプレートベース出力にフォールバック |
| レスポンス遅延（>3秒） | タイムアウト → テンプレート出力 |
| 出力パースエラー | 再試行（最大2回）→ テンプレート出力 |
| APIレート制限 | 指数バックオフ → テンプレート出力 |

### 5.2 ブランチ命名規則

| ブランチ種別 | 命名規則 | 例 |
|-------------|---------|-----|
| メイン | `main` | main |

### 5.3 コミットメッセージ規則

```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type:**

| タイプ | 説明 |
|--------|------|
| feat | 新機能 |
| fix | バグ修正 |
| docs | ドキュメント |
| style | フォーマット（コード動作に影響なし） |
| refactor | リファクタリング |
| perf | パフォーマンス改善 |
| test | テスト追加・修正 |
| chore | ビルド、ツール設定 |

**例:**

```

- CSVパースとバリデーションを実装
```

## 6. CI/CD パイプライン

### 6.1 CI パイプライン

```yaml
# GitHub Actions
name: CI

on:
  pull_request:
    branches: [develop, main]

jobs:
  lint:
    - ESLint / Ruff
    - Prettier / Black
    - TypeScript型チェック

  test:
    - ユニットテスト (Jest / pytest)
    - カバレッジレポート (80%以上必須)

  security:
    - 依存関係脆弱性スキャン (Dependabot)
    - SAST (CodeQL)

  build:
    - Dockerイメージビルド
    - イメージ脆弱性スキャン (Trivy)
```

### 6.2 CD パイプライン

```
[develop マージ]
    │
    ▼
[CI 実行] ──失敗──▶ 通知 → 修正
    │
    成功
    │
    ▼
[Development デプロイ] (自動)
    │
    ▼
[Staging デプロイ] (手動トリガー)
    │
    ▼

    │
    ▼

    ▼

    │
    ├─ 成功 ──▶ 完了

    └─ 失敗 ──▶ 自動ロールバック
```

## 7. リリースフロー

### 7.1 リリースサイクル

| 種別 | 頻度 | 内容 |
|------|------|------|
| 定期リリース | 隔週（スプリント終了時） | 新機能、改善 |
| パッチリリース | 随時 | バグ修正 |
| ホットフィックス | 緊急時 | 重大バグ、セキュリティ |

### 7.2 リリース手順

#### 定期リリース

1. **リリース準備**（リリース前日）
   - release/x.x.x ブランチ作成
   - バージョン番号更新
   - CHANGELOG 更新
   - Staging デプロイ・最終確認

2. **リリース実行**（リリース当日）
   - release → main マージ（PR + 承認）
   - タグ作成（v1.0.0）
   - Production デプロイ
   - ヘルスチェック確認
   - release → develop マージ

3. **リリース後**
   - リリースノート公開
   - ユーザー通知（必要に応じて）
   - メトリクス監視強化（24時間）

#### ホットフィックス

1. main から hotfix/xxx ブランチ作成
2. 修正実施・テスト
3. main へ直接マージ（緊急承認）
4. 即時デプロイ
5. develop へバックマージ

### 7.3 バージョニング

**セマンティックバージョニング** (SemVer)

```
MAJOR.MINOR.PATCH

例: 1.2.3
```

| バージョン | 更新タイミング |
|-----------|--------------|
| MAJOR | 後方互換性のない変更 |
| MINOR | 後方互換性のある機能追加 |
| PATCH | バグ修正 |

## 9. セキュリティ

### 9.1 認証・認可

| 項目 | 実装 |
|------|------|
| 認証方式 | JWT (RS256) |
| トークン有効期限 | アクセストークン: 1時間、リフレッシュトークン: 7日 |
| パスワードポリシー | 8文字以上、英数記号含む |
| ログイン試行制限 | 5回失敗で30分ロック |
| セッション管理 | Redis（サーバーサイド無効化対応） |

### 9.2 データ保護

| 対象 | 保護方式 |
|------|---------|
| 通信 | TLS 1.3 |
| 保存データ | AES-256（RDS暗号化） |
| パスワード | bcrypt (cost=12) |
| 個人情報 | アクセスログ記録、最小権限原則 |

### 9.3 外部連携セキュリティ

| 連携先 | 認証方式 | 注意点 |
|--------|---------|--------|
| Google Sheets API | OAuth 2.0 | トークン暗号化保存 |
| **Anthropic API（Claude 3）** | APIキー | Secrets Manager管理、レート制限監視 |
| OpenAI API（予備） | APIキー | Secrets Manager管理 |

#### LLMセキュリティ対策

| 対策 | 説明 |
|------|------|
| APIキーローテーション | 90日ごとにキー更新 |
| プロンプトインジェクション対策 | 入力サニタイズ、出力検証 |
| センシティブデータ除外 | 個人情報を含まないプロンプト設計 |
| 利用ログ記録 | 監査目的で全LLM呼び出しをログ保存（90日） |
| コスト制御 | 月額上限アラート、異常利用検知 |

### 9.4 脆弱性対策

| 対策 | 実施頻度 |
|------|---------|
| 依存関係スキャン | CI毎（Dependabot） |
| コンテナスキャン | CI毎（Trivy） |
| SAST | CI毎（CodeQL） |
| ペネトレーションテスト | 年1回 |

## 10. バックアップ・DR

### 10.1 バックアップ

| 対象 | 方式 | 頻度 | 保存期間 |
|------|------|------|----------|
| RDS | 自動スナップショット | 日次 | 35日 |

| S3 | バージョニング | 継続的 | 無期限 |

### 10.2 DR（災害復旧）

| 項目 | 目標値 |
|------|--------|
| RTO（復旧時間目標） | 4時間 |
| RPO（復旧ポイント目標） | 1時間 |
| DR環境 | 別リージョン（東京 → 大阪） |

### 10.3 復旧手順

1. 障害検知・判定
2. DR発動判断（ステークホルダー承認）
3. DR環境のRDSをプライマリに昇格
4. DNSフェイルオーバー
5. 動作確認
6. ユーザー通知
