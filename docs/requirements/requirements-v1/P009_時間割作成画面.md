## 2. 設計思想：熟練者の行動パターンの再現

### 2.1 熟練スケジューラーの特徴（研究知見より）

| 特徴 | 熟練者の行動 | 初心者の行動 |
|------|------------|------------|
| 外部表現の活用 | 可視化ツール（ボード、カレンダー）を活用 | 頭の中で制約を管理しようとする |
| 戦略と戦術の分離 | 「何をするか」を決めてから「どうやるか」を考える | 場当たり的に配置を試行 |
| 反復的改善 | 暫定解から段階的に改善 | 一発で完璧を目指す |
| 問題の分解 | 難しい部分を特定し優先的に解決 | 全体を一度に解こうとする |

### 2.2 本システムでの再現方針

1. **可視化ファースト**: 常にカレンダービューで状況を把握
2. **2フェーズアプローチ**: まず実行可能解、次に最適化
3. **段階的改善**: 小さな改善を積み重ねる
4. **人間との協調**: エージェントが提案し、人間が判断

## 4. エージェントアーキテクチャ

### 4.1 設計原則

熟練スケジューラーの思考プロセスを4つの専門エージェントに分解。
Schedule Orchestrator（状態マシン）が全体を統括し、それぞれのエージェントが明確な役割を持ち、協調して時間割を作成する。

```
┌─────────────────────────────────────────────────────────────────┐
│                    Schedule Orchestrator                        │
│                      【状態マシン】                              │
│   全体の進行管理・フェーズ遷移・精度モニタリング・ユーザー対話     │
└─────────────────────────────────────────────────────────────────┘
        │              │              │              │
        ▼              ▼              ▼              ▼
┌───────────┐  ┌───────────┐  ┌───────────┐  ┌───────────┐
│ Analyst   │  │ Builder   │  │ Evaluator │  │ Advisor   │
│ Agent     │  │ Agent     │  │ Agent     │  │ Agent     │
│【ルール】  │  │【OR-Tools】│  │【ルール+LLM】│ │【LLM】    │
│           │  │           │  │           │  │           │
│ 問題分析   │  │ 配置構築   │  │ 品質評価   │  │ 改善提案   │
│ 難易度判定 │  │ 制約充足   │  │ 違反検出   │  │ トレードオフ│
│ 優先順位付 │  │ 割当実行   │  │ 説明生成   │  │ 対話応答   │
└───────────┘  └───────────┘  └───────────┘  └───────────┘
```

### 4.2 Schedule Orchestrator（全体統括）

**役割**: 4つのエージェントを統括し、フェーズ遷移・精度モニタリング・ユーザー対話を管理
**実装方式**: 状態マシン（ルールベース）

#### 技術方式の選定理由

| 観点 | ルールベースを選択した理由 |
|------|-------------------------|
| 決定論性 | 同じ状況で常に同じ遷移を保証する必要がある |
| 説明責任 | ユーザーに「なぜこう進むのか」を説明可能にする |
| パフォーマンス | 即座の状態遷移が必要（LLM呼び出しは不適切） |
| テスト容易性 | エッジケースの網羅的テストが可能 |

#### 責務一覧

| 責務 | 説明 | 実装方式 |
|------|------|---------|
| 状態遷移管理 | Phase1→Phase2の遷移判定 | 状態マシン |
| エージェント実行制御 | Analyst→Builder→Evaluator→Advisorの順序管理 | ルール |
| 改善ループ制御 | Phase2のイテレーション管理 | ルール |
| 精度モニタリング | スコア変化の追跡・終了判定 | ルール |
| ユーザー対話コーディネーション | 提案表示・判断受付の橋渡し | 状態マシン |
| エラーリカバリー | 失敗時のハンドリング・部分解保存 | ルール |

#### Phase1→Phase2 移行判定ロジック

```
Phase1完了判定:
├─ ハード制約違反あり？
│   └─ YES → Phase1失敗 → ユーザー判断待ち
│            （制約緩和 or 手動調整 or キャンセル）
├─ 全員配置完了？
│   └─ NO → 配置不可の生徒を報告 → ユーザー判断待ち
└─ 全条件クリア → Phase2へ自動遷移
```

| 判定条件 | 結果 | 次のアクション |
|---------|------|--------------|
| ハード制約違反あり | Phase1失敗 | 問題箇所をユーザーに報告、判断を待つ |
| 未配置生徒あり | Phase1未完 | 未配置理由をユーザーに報告、判断を待つ |
| 全員配置 + ハード制約充足 | Phase1成功 | Phase2へ自動遷移 |

#### 精度向上モニタリング

**監視メトリクス**:

| メトリクス | 計算式 | 目標値 | 用途 |
|----------|--------|--------|------|
| 1対2充足率 | (1対2コマ × 2) / 総生徒コマ数 × 100 | 85%以上 | メイン指標 |
| ソフト制約達成率 | 達成した制約数 / 設定された制約数 × 100 | 80%以上 | 副指標 |
| 講師負荷バランス | 1 - (標準偏差 / 平均コマ数) | 0.8以上 | サブ指標 |

**イテレーションごとの追跡**:
```
各イテレーション後:
├─ 現在スコアと前回スコアの差分を計算
├─ 改善幅 = 今回充足率 - 前回充足率
├─ 改善履歴を記録（傾向分析用）
└─ 終了条件をチェック
```

#### 改善ループ終了条件

| 終了条件 | 判定基準 | 最終状態 |
|---------|---------|---------|
| 目標達成 | 充足率 ≥ 85% | 成功 |
| 改善余地なし | 2回連続で改善幅 < 1% | プラトー到達 |
| 反復上限 | 10回のイテレーション完了 | 上限到達 |
| ユーザー確定 | ユーザーが「確定」を選択 | ユーザー判断 |

**終了時の動作**:
- Evaluatorに最終スコアを計算させる
- Advisorに最終サマリーを生成させる
- 時間割確定フローへ遷移

#### エラーリカバリー

| エラーシナリオ | 発生箇所 | 対応 |
|--------------|---------|------|
| 講師不足で配置不可 | Phase1 (Builder) | 原因を報告、ユーザーが制約緩和を選択可能 |
| ソルバータイムアウト | Phase1 (Builder) | 中間結果を保存、手動調整モードを提案 |
| LLM呼び出し失敗 | Phase2 (Advisor/Evaluator) | テンプレートベースにフォールバック |
| マスタデータ不整合 | 初期ロード | バリデーションエラーを表示、修正を促す |

### 4.3 各エージェントの役割

#### Analyst Agent（分析担当）

**役割**: 熟練者が最初に行う「問題の見極め」を担当
**実装方式**: ルールベース（集計・数値分析）

| 機能 | 説明 |
|------|------|
| 問題規模分析 | 講師数・生徒数・コマ数から問題の複雑度を判定 |
| ボトルネック特定 | 配置が困難な生徒・時間帯を事前に特定 |
| 優先順位付け | どの生徒/講師から配置すべきかを決定 |
| 実行可能性診断 | ハード制約だけで解が存在するか事前チェック |

**出力例**:
```
問題分析レポート:
- 規模: 中（講師25名、生徒80名、週320コマ）
- 難易度: やや高（理由: 土曜に需要集中、中受算数の講師不足）
- ボトルネック: 土曜3限（需要15、供給10）
- 推奨戦略: 土曜の難配置生徒を優先的に割り当て
```

#### Builder Agent（構築担当）

**役割**: 熟練者が行う「実際の配置作業」を担当
**実装方式**: OR-Tools CP-SAT（制約充足問題ソルバー）

| 機能 | 説明 |
|------|------|
| 優先順配置 | Analystの優先順位に従い順次配置 |
| 制約伝播 | 1つの配置が他に与える影響を即座に反映 |
| バックトラック | 行き詰まった場合に戻って再試行 |
| 部分解の保存 | 途中状態を保存し再開可能に |

**アルゴリズム選択**:
| 問題特性 | 選択アルゴリズム |
|---------|----------------|
| 小規模（講師15名以下） | 制約充足ソルバー（CSP） |
| 中規模（16〜35名） | 貪欲法＋局所探索 |
| 大規模（36名以上） | 階層的分解＋並列探索 |

#### Evaluator Agent（評価担当）

**役割**: 熟練者が行う「結果の確認・問題発見」を担当
**実装方式**: ルールベース + LLM（説明生成・背景分析）

| 機能 | 実装方式 | 説明 |
|------|---------|------|
| 品質スコア算出 | ルールベース | 充足率、ソフト制約達成率を計算（決定論的） |
| 違反検出 | ルールベース | ハード制約違反、ソフト制約違反をリスト化（確定的判定） |
| 問題分類 | ルールベース | 違反を種類・深刻度で分類 |
| 改善余地判定 | ルールベース | これ以上の改善が可能かを判定 |
| **説明生成** | **LLM** | 「なぜこの違反が発生しているか」を自然言語で説明 |
| **背景分析** | **LLM** | 問題の構造的原因（一時的 vs 構造的）を解説 |

**LLMによる説明生成の例**:
```
ルールベース検出: 「水曜3限で生徒Xに対して講師が1名のみ」

LLM説明:
「水曜3限は、中受算数を受講する生徒が3名集中しています。
 講師のシフト希望と科目適合性の制約により、
 生徒X向けに配置できる講師が1名に限定されました。

 背景: このパターンは土曜にも発生しやすく、
 中受経験講師の採用強化が根本的な解決策となります。」
```

**評価指標**:
| 指標 | 計算式 | 目標 |
|------|--------|------|
| 1対2充足率 | (1対2コマ × 2) / 総生徒コマ数 × 100 | 85%以上 |
| ソフト制約達成率 | 達成した制約数 / 設定された制約数 × 100 | 80%以上 |
| 講師負荷バランス | 1 - (標準偏差 / 平均コマ数) | 0.8以上 |
| 時間帯カバー率 | 埋まった枠数 / 利用可能枠数 × 100 | - |

#### Advisor Agent（助言担当）

**役割**: 熟練者が行う「改善策の検討・説明」を担当
**実装方式**: LLM

| 機能 | 説明 |
|------|------|
| 改善提案生成 | 現在の問題に対する解決策を複数提示（案A/B/C） |
| トレードオフ説明 | 各選択肢のメリット・デメリットを自然言語で説明 |
| 影響シミュレーション | 変更適用後の充足率・違反件数を予測 |
| 根拠説明 | 「なぜこの提案か」をコンテキストを踏まえて説明 |
| **対話応答** | ユーザーの質問（「なぜ？」「他の案は？」）に応答 |

**LLMを採用した理由**:

| 観点 | LLMのメリット |
|------|-------------|
| 自然言語生成 | ユーザーが理解しやすい説明を生成 |
| コンテキスト理解 | 現在の問題状況を踏まえた提案が可能 |
| 複合問題対応 | 複数の問題が絡み合う状況にも対応 |
| 柔軟な推論 | 固定パターンに縛られない提案 |
| 対話機能 | ユーザーの追加質問に応答可能 |

**提案例**（LLM生成）:
```
問題: 土曜3限で中受算数の田中さんが未配置

【LLM分析】
土曜3限では中受算数の生徒が集中しており、講師不足が発生しています。

提案A: 佐藤先生（中受経験あり）を土曜3限に追加配置
  - メリット: 田中さんを配置可能、1対2も実現可能
  - デメリット: 佐藤先生の連続コマが3→4に増加
  - 充足率への影響: 82% → 84%
  - 【補足】佐藤先生の火3限は講師が過剰配置気味のため、
          そちらを調整する方が教室全体のバランス改善に繋がります。

提案B: 田中さんを土曜2限に移動（△の時間帯）
  - メリット: 追加の講師調整不要
  - デメリット: 生徒の希望度が下がる（○→△）
  - 充足率への影響: 変化なし
  - 【補足】土曜2限は空きが多いため、他の生徒との1対2も組みやすくなります。

どちらを採用しますか？ご質問があればお気軽にどうぞ。
```

## 5. 2フェーズ最適化プロセス

### 5.1 フェーズ1: 実行可能解の構築（F081）

**目的**: ハード制約をすべて満たす解を見つける

```
┌─────────────────────────────────────────────────────────────┐
│ フェーズ1: 実行可能解の構築                                   │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 問題分析（Analyst）                                      │
│     └─→ ボトルネック特定、優先順位決定                        │
│                                                             │
│  2. 優先順配置（Builder）                                    │
│     └─→ 難しい生徒から順に配置、制約伝播                      │
│                                                             │
│  3. 実行可能性チェック（Evaluator）                           │
│     ├─→ 全員配置成功 → フェーズ2へ                           │
│     └─→ 配置不可あり → 問題報告、ユーザー判断要求              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 配置不可時の対応

| 状況 | エージェントの対応 |
|------|------------------|
| 講師不足 | 「中受算数を教えられる講師が土曜に不足しています」と報告 |
| 時間帯競合 | 「山田さんの希望時間帯に空きがありません」と報告 |
| NG制約 | 「鈴木さんはNG講師が多く、配置可能な講師がいません」と報告 |

### 5.2 フェーズ2: 反復的最適化（F082）

**目的**: ソフト制約を可能な限り満たし、品質を向上

```
┌─────────────────────────────────────────────────────────────┐
│ フェーズ2: 反復的最適化                                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────────────────────────────────────┐           │
│  │ 改善ループ（最大10回または改善なしで終了）      │           │
│  │                                              │           │
│  │  1. 評価（Evaluator）                         │           │
│  │     └─→ 現在のスコア、違反リスト作成          │           │
│  │                                              │           │
│  │  2. 改善提案（Advisor）                       │           │
│  │     └─→ 改善策を優先度順に提示               │           │
│  │                                              │           │
│  │  3. 適用判断                                  │           │
│  │     ├─→ 自動モード: 高確信度の提案を自動適用   │           │
│  │     └─→ 対話モード: ユーザーに確認            │           │
│  │                                              │           │
│  │  4. 変更適用（Builder）                       │           │
│  │     └─→ 選択された改善を実行                 │           │
│  │                                              │           │
│  └──────────────────────────────────────────────┘           │
│                                                             │
│  終了条件:                                                   │
│  - 目標スコア達成                                            │
│  - 改善余地なし（Evaluator判定）                             │
│  - ユーザーが「確定」を選択                                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 6. 機能詳細

### F080: 問題分析・診断

#### 入力

- 講師マスタ、生徒マスタ、希望データ、制約条件

#### 処理

1. Analyst Agentが問題を分析
2. 分析レポートを生成
3. 推奨戦略を提示

#### 出力

| 項目 | 説明 |
|------|------|
| 問題規模 | 小/中/大 |
| 難易度 | 低/中/高 + 理由 |
| ボトルネック一覧 | 特に配置が困難な箇所 |
| 推奨戦略 | どのような順序・方針で進めるべきか |
| 推定所要時間 | 処理完了までの見込み時間 |

│  │      │ 佐藤   │ 田中   │ 鈴木   │ 田中   │ 佐藤   │ 佐藤  │    │

## 9. 設計根拠

### 9.1 2フェーズアプローチを採用した理由

- 数理最適化の標準的手法（フェージビリティ→オプティマリティ）
- 熟練者も「まず全員を配置してから調整」という順序で作業する
- 問題が解けない場合の原因特定が容易

### 9.2 4エージェント構成の理由

- 熟練者の思考プロセスを明確に分離・再現
- 各エージェントの責務が明確で、デバッグ・改善が容易
- ユーザーへの説明が自然（「分析中」「構築中」「評価中」「提案中」）

### 9.3 可視化ファーストの理由

- 研究知見: 熟練者は外部表現（ボード等）を活用する
- ユーザーが状況を直感的に把握できる
- 問題箇所の発見と対処が迅速

### 9.4 人間との協調を重視した理由

- 完全自動化よりも「提案→判断」の方がユーザー満足度が高い
- ドメイン知識をユーザーが補完できる
- 最終責任は人間が持つべき

### 9.5 技術方式決定の根拠

#### 各コンポーネントの技術選択

| コンポーネント | 技術方式 | 選択理由 |
|--------------|---------|---------|
| **Orchestrator** | 状態マシン | 進行管理は決定論的であるべき。同じ状況で同じ遷移を保証し、ユーザーに説明可能にする必要がある |
| **Analyst** | ルールベース | 問題分析は数値的集計と閾値判定で十分。LLMによる計算ミスのリスクを回避 |
| **Builder** | OR-Tools | 制約充足問題はLLMが苦手。100%の制約遵守が必要な領域にはソルバーが必須 |
| **Evaluator** | ルール + LLM | スコア計算は決定論的であるべきだが、「なぜ問題か」の説明はLLMが得意 |
| **Advisor** | LLM | 自然言語での説明、複合問題への対応、対話機能でLLMの価値が最も高い |

#### LLMを使わない理由（Orchestrator/Analyst/Builder）

| 観点 | 説明 |
|------|------|
| 決定論性 | 同じ入力に対して同じ出力を保証する必要がある処理にLLMは不向き |
| 制約遵守 | LLMは「講師の連続コマ上限5なのに7コマ配置」のような制約違反を起こしうる |
| パフォーマンス | Phase1は10〜60秒の制限があり、LLM呼び出しのレイテンシは許容困難 |
| テスト容易性 | ルールベースは網羅的テストが可能、LLMは出力が非決定的でテスト困難 |

#### LLMを使う理由（Evaluator/Advisor）

| 観点 | 説明 |
|------|------|
| 自然言語生成 | 違反の背景説明、提案の根拠説明など、人間が理解しやすい表現が求められる |
| コンテキスト理解 | 複数の制約・状況を総合的に判断した提案が可能 |
| 柔軟な対応 | 固定パターンでは対応できない複合問題にも対処可能 |
| 対話機能 | 「なぜ？」「他の案は？」といったユーザーの追加質問に応答可能 |

#### パフォーマンス要件との整合性

| フェーズ | 時間制限 | LLM適用 |
|---------|--------|---------|
| フェーズ1（実行可能解構築） | 10〜60秒 | ❌ リアルタイム性重要、ソルバー必須 |
| フェーズ2（改善ループ） | 1ループ5秒 | ✅ LLM呼び出し（500ms〜1秒）は許容範囲 |

AdvisorとEvaluatorのLLM呼び出しはフェーズ2で実行されるため、レイテンシは許容可能。

### 10.3 状態保存とUndo/Redo

#### 基本仕様

- 各フェーズの途中状態を自動保存（LocalStorage + サーバー同期）
- ブラウザを閉じても再開可能
- Undo/Redo対応（最大50ステップ）

#### 同時編集時の整合性

| 状況 | 対応 |
|------|------|
| Undo中に他ユーザーがマスタを編集 | Undo不可。「マスタデータが更新されました。最新の状態で続行しますか？」と表示 |
| 同一タームを複数ユーザーが編集 | 楽観的ロック（バージョン番号）で検出。後から保存しようとした方に警告 |
| バージョン競合発生時 | 「他のユーザーが編集中です。上書きしますか？ / 変更を破棄しますか？」の選択肢を表示 |

#### 楽観的ロックの実装

```
保存時:
1. クライアントが保持しているversion_numberをサーバーに送信
2. サーバーはDBのversion_numberと比較
3. 一致 → 保存成功、version_number + 1
4. 不一致 → コンフリクトエラーを返却、クライアントに選択肢を提示
```
