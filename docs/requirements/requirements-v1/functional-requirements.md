## 3. ダッシュボード機能（P002）

### F010: ダッシュボード表示

教室長向けのメイン画面。以下のウィジェットを表示:
- 充足率サマリー（F011）
- 人員状況ヒートマップ（F012）
- 科目別カバー率（F013）
- 需給バランス（F014）
- 通知・アラート（F015）

### F011: 充足率サマリー表示

**用語定義**:
- **1対2コマ数**: 1人の講師が2人の生徒を同時に教えるコマの数
- **1対1コマ数**: 1人の講師が1人の生徒を教えるコマの数

| 項目名 | 計算式 |
|--------|--------|
| 充足率 | (1対2コマ数 × 2) / (1対2コマ数 × 2 + 1対1コマ数) × 100 |

**エッジケースの扱い**:
- 生徒が0人（分母が0）: 表示「-」（計算対象外）
- 未配置の生徒: 分母・分子ともに含めない

### F012: 人員状況ヒートマップ表示

曜日×限のマトリクスで需給状況を色分け表示。

#### 需給差の定義

各時間枠（曜日×限）における「講師が提供可能な授業枠数」と「生徒の受講希望人数」のバランスを示す指標。

| 項目 | 計算式 |
|------|--------|
| 需給差 | 供給枠数 - 需要人数 |
| 供給枠数 | その枠で勤務可能な講師数 × 2（1対2形式想定） |
| 需要人数 | その枠で受講可能（○または△）の生徒数 |

**解釈**: 「その枠であと何人の生徒を受け入れ可能か」を示す。

| 需給差 | ステータス | 表示色 |
|--------|-----------|--------|
| >= 2 | 余裕あり | 緑 |
| 1 | 適正 | 黄緑 |
| 0 | ギリギリ | 黄 |
| < 0 | 不足 | 赤 |

**データソース**:
- 供給: 講師シフト希望（◯の講師数）
- 需要: 生徒受講希望（○/△の生徒数）

### F013: 科目別カバー率表示

科目ごとに「その科目の授業を十分に提供できる講師がいるか」を示す指標。

#### この指標で知りたいこと

「数学を週40コマ分教えてほしい生徒がいるけど、数学を教えられる講師は足りている？」という問いに答える。

#### 計算の考え方

##### ステップ1: 需要を集計する

生徒マスタの「科目別コマ数」から、科目ごとの需要コマ数を合計する。

**例**:
| 生徒 | 科目別コマ数 |
|------|-------------|
| 生徒A | 数学:2 |
| 生徒B | 数学:1, 英語:2 |
| 生徒C | 英語:3 |

→ 数学の需要 = 2 + 1 = **3コマ**
→ 英語の需要 = 2 + 3 = **5コマ**

##### ステップ2: 供給を集計する（按分方式）

講師マスタの「最大コマ/週」から供給を計算する。ただし、1人の講師が複数科目を教えられる場合、そのまま足すと二重カウントになる問題がある。

**悪い例（二重カウント）**:
| 講師 | 教えられる科目 | 最大コマ/週 |
|------|--------------|------------|
| 講師X | 数学, 英語 | 4 |

→ 数学の供給 = 4コマ、英語の供給 = 4コマ？
→ 合計8コマ供給できることになるが、講師Xは週4コマしか働けない。**おかしい。**

**按分方式（正しい計算）**:

講師Xの4コマを、数学と英語の需要の比率で分ける。

- 数学の需要 = 3コマ
- 英語の需要 = 5コマ
- 合計 = 8コマ

講師Xの供給の配分:
- 数学への供給 = 4 × (3 ÷ 8) = **1.5コマ**
- 英語への供給 = 4 × (5 ÷ 8) = **2.5コマ**

これなら合計4コマで、講師Xの実際の働ける量と一致する。

##### ステップ3: カバー率を計算する

```
科目別カバー率 = 供給コマ数 ÷ 需要コマ数 × 100
```

**例**:
- 数学: 1.5 ÷ 3 × 100 = **50%**（不足）
- 英語: 2.5 ÷ 5 × 100 = **50%**（不足）

##### ステップ4: 時間帯の制約も確認する

上記の計算だけでは「講師と生徒の希望時間帯が合っているか」が分からない。

例えば、数学を教えられる講師が月曜しか出勤できないのに、数学を受けたい生徒が全員土曜希望だと、実際にはマッチできない。

そこで、各時間枠ごとに「その科目を教えられる講師がいるか」も確認し、時間帯が合わない場合はカバー率を下げる。

#### 計算式（まとめ）

```
科目別カバー率 = min(供給充足度, 時間帯充足度) × 100

供給充足度 = 按分した供給コマ数 ÷ 需要コマ数

時間帯充足度 =
  全枠の「その科目を教えられる講師数 × 2」の合計 ÷
  全枠の「その科目を受講希望する生徒数」の合計
```

2つの充足度のうち、低い方を採用する。「講師の数が足りない」「時間帯が合わない」のどちらかでも問題があればカバー率は下がる。

#### 表示

| カバー率 | ステータス | 表示色 |
|---------|-----------|--------|
| 100%以上 | 充足 | 緑 |
| 70〜99% | やや不足 | 黄 |
| 70%未満 | 不足 | 赤 |

#### データソース

- 需要: 生徒マスタの「科目別コマ数」
- 供給: 講師マスタの「指導可能科目」「最大コマ/週」
- 時間帯: 講師シフト希望、生徒受講希望

## 4. マネジメントダッシュボード機能（P003）

| 項目名 | 説明 |
|--------|------|

### F023: 問題教室アラート

充足率が閾値以下、または前月比で大幅低下した教室を強調表示

| 条件 | アラートレベル | 表示 |
|------|--------------|------|
| 充足率60%未満 | 危険 | 赤バッジ |
| 前月比10ポイント以上低下 | 危険 | 赤バッジ |
| 充足率60〜79% | 警告 | 黄バッジ |
| 未確定の時間割が期限超過 | 警告 | 黄バッジ |

### F024: 教室比較

| 比較項目 | 表示内容 |
|---------|---------|
| 充足率 | 各教室の充足率を横並びで表示 |
| 前月比 | 各教室の増減を横並びで表示 |
| 講師数/生徒数 | 人員状況の比較 |
| 充足率推移 | 複数教室の推移を重ねたグラフ |

**処理**: チェックボックスで2〜4教室を選択 → 比較ボタン → 比較モーダル表示

| システム項目 | 説明 | 自動検出キーワード |
|-------------|------|------------------|

| 曜日 | 希望曜日 | "曜日" |
| 時限 | 希望時限 | "時限", "コマ", "時間" |

1. スプレッドシートのヘッダー行（1行目）を取得
2. 各列名に対してキーワードマッチングを実行

- プレビュー表示で取り込み結果を事前確認

F041と同じ項目を編集。変更時は新バージョンを作成（16章参照）。

**制限**: 最大30件/回

```csv
氏名,性別,指導可能科目,指導可能学年,最小コマ,最大コマ,最大連続コマ,大学ランク,中受経験,高受経験
```

登録済み講師データをCSV形式でエクスポート。

**出力項目**:
```csv
講師ID,氏名,性別,指導可能科目,指導可能学年,最小コマ,最大コマ,最大連続コマ,大学ランク,中受経験,高受経験,ステータス
```

**エクスポート範囲**:
- 全件: 有効な全講師データ
- フィルター適用後: 現在の検索・フィルター条件に該当するデータのみ

## 10. 時間割生成機能（P009）

### 10.1 設計思想：熟練スケジューラーの行動再現

熟練者の特徴を4つの専門エージェントで再現:
- **可視化ファースト**: 常にカレンダービューで状況を把握
- **2フェーズアプローチ**: まず実行可能解、次に最適化
- **段階的改善**: 小さな改善を積み重ねる
- **人間との協調**: エージェントが提案し、人間が判断

### 10.2 エージェントアーキテクチャ

| エージェント | 役割 | 機能 | 実装方式 |
|-------------|------|------|---------|
| Analyst Agent | 問題の見極め | 問題規模分析、ボトルネック特定、優先順位付け | ルールベース |
| Builder Agent | 配置作業 | 優先順配置、制約伝播、バックトラック | OR-Tools CP-SAT |
| Evaluator Agent | 結果確認 | 品質スコア算出、違反検出、改善余地判定 | ルールベース |
| Advisor Agent | 改善検討 | 改善提案生成、トレードオフ説明、影響シミュレーション | ルールベース + テンプレート |

※ v1では全エージェントをルールベース/アルゴリズムベースで実装。LLMは使用しない。
詳細な技術仕様はP009_時間割作成画面.mdの10章を参照。

### 10.3 2フェーズ最適化プロセス

**フェーズ1: 実行可能解の構築**
- 目的: ハード制約をすべて満たす解を見つける
- Analystがボトルネック特定 → Builderが優先順配置 → Evaluatorが検証

**フェーズ2: 反復的最適化**
- 目的: ソフト制約を可能な限り満たし、品質を向上
- Evaluatorが評価 → Advisorが提案 → ユーザー判断 → Builder適用（最大10回ループ）

### F080: 問題分析・診断

Analyst Agentが問題を分析し、規模・難易度・ボトルネック・推奨戦略を提示。

### F081: フェーズ1実行（実行可能解構築）

ハード制約を満たす初期解を構築。配置不可の場合は原因を報告。

### F082: フェーズ2実行（反復的最適化）

自動モード/対話モードを選択可能。Advisorが改善提案を行い、段階的に品質向上。

### F083: カレンダービュー（メイン操作画面）

熟練者が使う「ボード」を再現。すべての操作はこのビューを中心に行う。
- セル表示: 1対2コマ（緑）、1対1コマ（黄）、未配置（オレンジ）、問題あり（赤）
- インタラクション: クリック詳細、ダブルクリック編集、ドラッグ&ドロップ移動

### F084: 問題箇所ハイライト・ガイド付き解決

問題セルをクリックするとAdvisor Agentが解決策を複数提示。各案のメリット・デメリット・影響を明示。

### F085: 手動調整（ドラッグ&ドロップ）

コマをドラッグ → 移動可能先を色分け表示（緑=OK、黄=ソフト違反、赤=ハード違反）→ ドロップ → 即時スコア再計算

### F086: 時間割確定

確定前にハード制約違反チェック。draft → confirmed へステータス変更。確定後は直接編集不可。

### F087: PDF/CSV出力

| 出力種別 | 内容 | 用途 |
|---------|------|------|
| 全体表 | 週間カレンダー形式 | 教室掲示用 |
| 講師別 | 講師ごとの週間スケジュール | 講師への配布 |
| 生徒別 | 生徒ごとの週間スケジュール | 保護者への配布 |
| CSV | 全データ | 外部連携用 |

## 16. データバージョン管理

### 16.1 バージョン管理が必要なデータ

| データ | 変更頻度 | バージョン管理 |
|--------|---------|--------------|
| 講師マスタ | 中 | **必須** |
| 生徒マスタ | 中 | **必須** |
| 時間割 | 高 | **必須** |
| 希望データ | 中 | 推奨 |
| 制約条件 | 低 | 推奨 |

### 16.2 マスタのバージョン管理

```
Teacher / Student:
  - teacher_id / student_id: ID（不変）
  - version_id: バージョンID（例: T001_v3）
  - version_number: バージョン番号
  - is_current: 現在有効なバージョンか
  - valid_from / valid_to: 有効期間
  - change_reason_type: 変更理由種別（enum）
  - change_reason_note: 変更理由補足（自由入力、任意）
```

**変更理由種別（enum）**:
- 講師: SCHEDULE_CHANGE / SUBJECT_CHANGE / GRADE_CHANGE / NG_CHANGE / STATUS_CHANGE / INITIAL / OTHER
- 生徒: COURSE_CHANGE / GRADE_UP / PREFERENCE_CHANGE / GOAL_CHANGE / STATUS_CHANGE / INITIAL / OTHER

※ `OTHER` を選択した場合は `change_reason_note` の入力を必須とする。詳細は共通機能.mdを参照。

**更新時の処理**: 新バージョン作成 → 旧バージョンのis_current=false, valid_to=現在日時

### 16.3 時間割バージョン管理

```
Schedule:
  - schedule_id: 時間割ID
  - term_id: タームID（例: 2025-04）
  - version: バージョン番号
  - status: draft / confirmed / archived
  - parent_version: 派生元バージョン
  - master_snapshot: 作成時のマスタバージョン参照
  - generation_config: 生成時の設定（再現性のため）
```

**ステータス遷移**:
- draft: 作成中。削除可能、確定可能
- confirmed: 確定済み。直接編集不可、派生して新バージョン作成
- archived: 過去ターム。参照のみ

**確定済み時間割の保護**:
- 参照・PDF出力: 可能
- 直接編集: 不可
- 振替対応: 新バージョンを作成して反映

## 17. バリデーションルール

### 学年と科目の整合性

| 学年カテゴリ | 設定可能な科目 |
|------------|--------------|
| 小学生（小1〜小6） | 小学英語、公立算数、中受算数、小学国語、小論、公立理科、中受理科、公立社会、中受社会 |
| 中学生（中1〜中3） | 中学英語、公立数学、私立数学、中学国語、中学理科、中学社会 |
| 高校生（高1〜高3） | 高校12英語、高3英語、高校国語、数1A、数ⅡB、数Ⅲ、生物、化学、物理、世界史、日本史、地理 |

### NG関係の相互性

- 講師側でNG生徒を設定 → 生徒側のNG講師に自動追加（逆も同様）
- 片方を削除 → 相互関係も自動削除

### コマ数の整合性

- 講師: 最小コマ ≤ 割り当てコマ ≤ 最大コマ
- 生徒: 科目別コマ数の合計 = 週の受講コマ数
- 連続コマ: 設定値を超えない
